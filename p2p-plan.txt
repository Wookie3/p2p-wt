P2P Walkie Talkie App - Project Plan

=====================================

Tech Stack
----------
Frontend:
- Next.js 14+ (App Router) - Modern React framework with SSR
- TypeScript - Type safety
- Tailwind CSS - Sleek, modern styling
- shadcn/ui - Beautiful, accessible UI components

Backend/Infrastructure:
- Supabase - Database, Auth, and Realtime
- PeerJS - WebRTC wrapper for P2P voice communication
- Supabase Realtime - Signaling for WebRTC connections

Additional Tools:
- Zustand - Lightweight state management
- react-hot-toast - Beautiful notifications

=====================================

Database Schema (Supabase)
--------------------------

-- Profiles (extends Supabase auth)
profiles (
  id: uuid (references auth.users),
  username: text (unique),
  is_online: boolean,
  peer_id: text (for WebRTC),
  created_at: timestamp
)

-- Contacts/Friends
contacts (
  id: uuid,
  user_id: uuid (references profiles),
  contact_id: uuid (references profiles),
  created_at: timestamp,
  CONSTRAINT unique_contact UNIQUE (user_id, contact_id)
)

-- Friend Requests
friend_requests (
  id: uuid,
  from_user_id: uuid (references profiles),
  to_user_id: uuid (references profiles),
  status: enum ('pending', 'accepted', 'rejected'),
  created_at: timestamp
)

-- No call history at this stage

=====================================

Configuration Answers
---------------------
1. Authentication: Email/password only
2. Anonymous session duration: 12 hours
3. Audio quality: Medium-low (optimized for simple radio communication, bandwidth-efficient)
4. Contact limit: 100 contacts per user
5. Call history: Not needed at this stage

Core Features
-------------

1. User Authentication
   - Anonymous users get generated username (verb+animal)
   - Login via Supabase with email/password
   - Session persistence for logged-in users only
   - Anonymous users have 12-hour ephemeral sessions

2. Username Generation
   - Random verb + animal combinations (e.g., "RunningDuck", "SleepyLion")
   - Check uniqueness before assignment
   - Option to regenerate username (anonymous only)

3. Contact System
   - Search users by exact username
   - Send friend requests
   - Accept/reject requests
   - View contact list with online status
   - Maximum 100 contacts per user
   - Real-time status updates via Supabase Realtime

4. P2P Voice Communication
   - Push-to-talk interface (hold to transmit)
   - WebRTC via PeerJS for direct P2P audio
   - Supabase Realtime for signaling/negotiation
   - Multiple simultaneous conversations (tabs per contact)
   - Audio codec: Opus at medium-low bitrate (16-24 kbps)
   - Visual feedback when transmitting/receiving

5. UI/UX
   - Dark mode by default (sleek look)
   - Animated transitions
   - Push-to-talk button with press-and-hold animation
   - Contact status indicators (online/offline)
   - Toast notifications for requests/messages

Technical Considerations
------------------------

P2P Signaling Flow:
1. User A initiates call to User B
2. User A generates PeerJS offer
3. Send offer via Supabase Realtime to User B
4. User B accepts and generates answer
5. Send answer back via Supabase Realtime
6. Establish WebRTC connection
7. Exchange audio streams directly (peer-to-peer)

Push-to-Talk Logic:
- Mouse/Touch Down → Start MediaRecorder → Capture audio chunks
- Mouse/Touch Up → Stop recording → Send via PeerJS data channel
- Receive → Decode chunks → Play via Web Audio API

Audio Configuration:
- Codec: Opus
- Bitrate: 16-24 kbps (medium-low)
- Sample rate: 16 kHz (sufficient for voice)
- Frame duration: 20-60ms (adjustable for latency/quality tradeoff)

Username Format:
- Verbs: Running, Sleeping, Happy, Angry, Fast, Slow, Silent, Loud, Brave, Calm, etc.
- Animals: Duck, Lion, Bear, Fox, Eagle, Wolf, Cat, Dog, Tiger, Owl, etc.
- Example: "RunningDuck", "SleepyFox", "HappyEagle", "AngryTiger"

State Management (Zustand):
- Current user profile
- Contact list (max 100)
- Active calls
- Friend requests
- Online status of contacts
- Anonymous session timestamp

RLS Policies:
- Users can only read/write their own profiles
- Users can only read/write their own contacts
- Users can only read/write their own friend requests
- Public read access for usernames (for search)
